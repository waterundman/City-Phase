# 城市相 · CityPhase — 项目分析与迭代计划

## 一、项目特点总结

### 1. 项目定位
**城市相**是一个 Blender 原生插件，核心命题是"几何优先"——通过程序化手段从 OSM 地理数据或参数化输入生成城市白膜（白色体量模型），并预制四套针对亚洲高密度城市美学优化的渲染管线。

### 2. 核心设计哲学
- **白膜表达语言**：消除材质噪声，让城市体量关系、密度节奏和空间句法本身成为视觉主角
- **亚洲类型学驱动**：六种建筑类型（塔楼、裙房+塔楼、板楼、老居民楼、综合体、工业厂房）覆盖现代亚洲高密度城市 90% 以上的体量类型
- **双模式数据管线**：OSM 真实数据导入 + 程序化参数生成，互为补充

### 3. 技术架构特点
- **分层生成管线**：路网生成 → DCEL 街区提取 → 地块分割 → 高度分配 → 单体建筑生成，五层管线依次传递
- **BMesh 几何框架**：核心几何操作（extrude_face、scale_face_xy、rotate_face_z）构成统一的建筑生成基础
- **高度双变量模型**：离中心距离决定基准高度 + 高斯噪声场决定相邻建筑高度相关性
- **四套渲染预设**：Arch White（白膜分析）、Golden Hour（黄昏金切）、Neon Rain（霓虹雨夜）、Iso Clear（等轴晴空）

### 4. 当前状态
| 维度 | 状态 |
|------|------|
| 设计文档 | 完整（JSX 交互式文档 + Word 工程文档，双份互证） |
| 代码实现 | 仅单体建筑生成器（3 种形态），可独立在 Blender 中运行 |
| 插件骨架 | 未实现（仅在设计文档中规划了目录结构） |
| OSM 集成 | 未实现 |
| 渲染管线 | 未实现 |
| 路网/街区/地块生成 | 未实现 |

### 5. 项目优势
- 设计文档极其详尽，算法逻辑、数据结构、UI 设计、开发阶段均有清晰规划
- 单体建筑生成器代码质量高，BMesh 操作封装良好，参数化程度高
- 类型学归纳精准（上海/东京 CBD 形态观察），有明确的审美指向
- 渲染管线预设具有差异化竞争力（非通用城市生成器，而是专注亚洲城市美学）

### 6. 项目风险
- Overpass API 频率限制，需本地缓存层
- 大城市 OSM 数据可能包含十万级节点，需 BMesh 高效构建
- Blender 不支持真正异步，网络请求需 modal() + timer 模式
- 第三方依赖（requests 等）在 Blender 插件中的分发问题

## 二、迭代计划

### 迭代 0：插件骨架（Phase 01 落地）
**目标**：最小可运行版本，在 Blender 中看到第一个可交互生成的单体建筑
**预计工期**：1–2 周

| 优先级 | 任务 | 产出 | 依赖 |
|--------|------|------|------|
| P0 | `__init__.py` 注册结构 + bl_info | 可安装启用的 Blender 插件 | 无 |
| P0 | `properties.py` 最小 PropertyGroup | base_w, base_d, height, typology, seed 五个属性 | 无 |
| P0 | `panels/panel_main.py` 基础 N-Panel | 含参数输入区 + 生成按钮的侧面板 | properties.py |
| P0 | 迁移 `cityp_building_gen.py` → `generators/building_gen.py` | 适配插件架构，从 main() 改为函数调用 | 无 |
| P0 | `operators/op_generate.py` 主 Operator | 读取 properties → 调用 building_gen → 自动应用白膜材质 | properties.py, building_gen.py |
| P1 | 添加 typology 下拉枚举 UI | 三种形态可选 | properties.py |

**里程碑**：安装插件 → 打开 N-Panel → 调整参数 → 点击生成 → 场景原点出现参数化建筑白膜

### 迭代 1：程序化城市生成（Phase 02 前半）
**目标**：不依赖 OSM，纯程序化生成一个完整城市的白膜
**预计工期**：2–3 周

| 优先级 | 任务 | 产出 | 依赖 |
|--------|------|------|------|
| P0 | `generators/city_layout.py` 路网生成 | 正交网格 + 有机扰动，输出 RoadGraph | 迭代 0 |
| P0 | DCEL 街区提取算法 | 从路网图提取有界面集（城市街区多边形） | city_layout.py |
| P0 | 街区 → 地块分割（退线 + 递归轴向分割） | 每个街区内生成建筑地块 Polygon | DCEL 提取 |
| P0 | 高度分配算法（双变量模型） | 每个地块分配基准高度 | 地块分割 |
| P0 | `batch_place_buildings()` 批量生成 | 遍历地块，调用 building_gen 实例化所有建筑 | building_gen.py, 高度分配 |
| P1 | 种子树系统 | 城市级种子派生每栋建筑独立种子，保证可复现 | batch_place_buildings |
| P1 | Collection 层级组织 | 建筑/道路分 Collection 管理 | batch_place_buildings |

**里程碑**：输入网格密度 + 种子 + 范围 → 生成包含路网 + 数十栋建筑的完整城市白膜

## 三、关键建议

1. **严格遵循 Phase 01 最小化原则**：先跑通"安装插件 → 调参 → 生成单体建筑"的完整链路，不要一开始就做 OSM
2. **building_gen.py 需要重构**：当前是脚本模式（全局参数 P + main()），需改为函数式 API，接受参数返回 mesh 数据
3. **OSM 缓存层必须尽早设计**：Overpass API 有频率限制，本地 JSON 缓存的存储路径和失效策略需要在迭代 2 之前就确定
4. **DCEL 算法是最大技术风险**：从路网图提取街区多边形是算法最复杂的一环，建议先用 shapely 差集操作做 MVP，再考虑优化
5. **渲染管线可以并行开发**：四个预设互相独立，可以在迭代 3 进行期间就开始编写
