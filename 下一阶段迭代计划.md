## 一、当前算法瓶颈分析

### 1.1 路网生成 — 过于规则化

**现状**：正交网格 + 均匀扰动，生成效果类似规划新区的 CAD 图纸，缺乏真实城市的有机感。

**问题**：
- 无环状道路（缺少城市环路、次干路环线）
- 无放射状道路（缺少从中心向外辐射的主轴）
- `sub_grid_spacing` 参数已定义但**从未使用**
- 无 waterfront/terrain-aware 路由
- 交叉口全部为十字，无 T 字、五叉等复杂交叉口

### 1.2 街区提取 — 右转法局限

**现状**：DCEL 右转法，O(n²) 复杂度，仅适用于简单平面图。

**问题**：
- 复杂交叉口会产生碎小伪街区
- 无 concave block 处理
- 街区形状与真实城市差距大（真实街区受历史、地形影响呈不规则形状）

### 1.3 地块分割 — 简单到失真

**现状**：质心收缩退线 + 随机轴向二分递归。

**问题**：
- 退线算法对非凸多边形会产生自交
- 仅有 0/90° 轴向分割，无斜向分割
- 无法生成 L 形、U 形地块（真实城市中常见）
- 未考虑临街面朝向（建筑应优先朝向街道）

### 1.4 建筑生成 — 缺乏建筑学细节

**现状**：BMesh 拉伸 Box，仅 7 种简单形态。

**问题**：
- 无立面细节（窗格、阳台、雨棚、空调外机）
- 无屋顶多样性（平屋顶、坡屋顶、穹顶、退台花园）
- 无入口/大堂体量突出
- 材质纯白色，无 grime/weathering/边缘 dirty
- 无裙房商业层通透感（玻璃幕墙 vs 实墙对比）

### 1.5 高度分配 — 与真实城市脱节

**现状**：线性/bimodal 距离衰减 + 高斯噪声。

**问题**：
- 真实城市高度分布不是同心圆，而是**沿地铁线/主轴的走廊式分布**
- 无 waterfront premium（临水建筑通常更高）
- 无 transit-oriented density spike（地铁站上方通常是最高建筑）
- 无历史保护区高度限制（老城核心区反而矮）

### 1.6 渲染管线 — 缺乏视觉层次

**现状**：4 套基础灯光+材质预设。

**问题**：
- 无程序化 grime 纹理（白膜太干净，缺乏城市质感）
- 无 AO（Ambient Occlusion）预计算，建筑间缺乏接触阴影
- 无边缘 chamfer/bevel，所有边都是锐角，缺乏体量感
- 无雾效/大气透视，远景缺乏层次
- 无夜间自发光窗格（霓虹雨夜管线应有 but 缺失）

## 二、核心算法研究方向

### 阶段 A：路网算法升级（2-3 周）

#### A1. 混合路网模式

研究并实现** radial + grid + organic 混合路网**：

```
模式 1: 纯网格（Grid）
  → 适合北美/中国新城
  
模式 2: 放射+环路（Radial-Ring）
  → 适合巴黎、莫斯科、北京二环内
  → 算法：同心圆道路 + 放射状主轴
  
模式 3: 有机生长（Organic / L-System）
  → 适合欧洲老城、东京小巷
  → 算法：L-System / 张量场引导 / Voronoi 细分
  
模式 4: 水网型（Waterfront）
  → 适合威尼斯、上海外滩、香港中环
  → 算法：沿水边平行道路 + 垂直岸线的小巷
```

#### A2. 张量场引导路网（Tensor Field Guided Roads）

参考论文：*"Procedural Modeling of Cities" (Parish & Müller, 2001)*

核心思想：
- 定义一个张量场（主要方向 + 次要方向）
- 道路沿张量场的特征方向延伸
- 遇到已有道路时停止（碰撞检测）
- 在交叉口根据张量方向决定继续/转弯/终止

视觉收益：生成类似曼哈顿网格、巴黎放射、伦敦有机混合的真实路网。

#### A3. 次级路网细分

**利用现有的 `sub_grid_spacing` 参数**：

```
阶段 1: 生成主干道（main_spacing = 300m）
阶段 2: 在每个街区内细分次干道（sub_spacing = 90m）
阶段 3: 在次干道街区再细分小巷（alley_width = 6m）
```

实现：将当前路网生成器改为**递归分层**，每层独立生成并叠加。

```python
    # 将立面分割为 grid
    cols = int(face_width / 3.6)  # 标准柱距 3.6m
    rows = int(face_height / floor_height)
    
    for row in range(rows):
        for col in range(cols):
            # 决定此处是 window / solid / balcony
            if style == "modern" and row > 1:  # 底层商业
                place_window(bm, ...)
            elif style == "old_res" and col % 2 == 0:
                place_window(bm, ...)
            else:
                place_solid(bm, ...)
```

视觉收益：同一建筑立面上有窗格、实墙、阳台的交替，白膜不再是一面白墙。

## 五、关键技术决策

### 决策 1：是否引入外部几何库？

- **shapely**（Python）：功能完善，但安装复杂（需 GEOS C++ 库），Blender 插件分发困难
- **blender.geometry.nodes**（节点）：Blender 原生，但无法 Python 精细控制
- **自行实现**：零依赖，可控性强，推荐

**结论**：自行实现多边形偏移、布尔运算等核心几何算法。

### 决策 2：Facade Grammar 的实现方式？

- **选项 A**：BMesh 直接建模窗格（面数爆炸，1000 建筑 × 100 窗格 = 10 万面）
- **选项 B**：Geometry Nodes 程序化窗格（Blender 4.x 推荐，GPU 实例化）
- **选项 C**：材质 shader 模拟窗格（通过 UV + noise 在平面上画窗格，零几何开销）

**结论**：
- 近景 LOD 2-3：Geometry Nodes 实例化窗格
- 远景 LOD 0-1：材质 shader 模拟
- 这样平衡了视觉效果与性能

### 决策 3：夜间发光窗格的实现？

- **选项 A**：真实几何发光面（Mesh 面 + 自发光材质）
- **选项 B**：Emission shader 在材质层面（通过 UV mask 控制哪些窗格发光）

**结论**：选项 B。通过一个 Noise Texture → ColorRamp → Emission Shader 的节点组，在材质层面实现随机窗格发光，零几何开销。
